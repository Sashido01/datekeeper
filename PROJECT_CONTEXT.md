## 1. ОБЩАЯ ИНФОРМАЦИЯ

### 1.1. Название проекта
**DateKeeper** — приватный офлайн-помощник для хранения и напоминания важных дат.

### 1.2. Уникальная ценность (UVP)
- **Privacy-first:** все данные хранятся локально, никакой телеметрии, открытый код (MIT).
- **Offline-first:** приложение полностью работает без интернета (кроме опциональной синхронизации).
- **Подарочный фонд:** возможность сохранять идеи подарков и получать о них напоминания.
- **Кроссплатформенность:** Windows, Linux, Android, iOS (поэтапно).

### 1.3. Целевая аудитория
1. Privacy-энтузиасты, технари (Linux/Windows).
2. Занятые профессионалы.
3. Семьи.
4. Обычные пользователи (после стабилизации).

## 2. АРХИТЕКТУРА

### 2.1. Общая модель
Используется **упрощённая Clean Architecture** с тремя слоями:

```
Presentation (UI, Riverpod)
      ⇅
   Domain (Use Cases, Entities, Repository Interfaces)
      ⇅
    Data (Repositories, Data Sources: local DB, WebDAV, Platform APIs)
```

- **Presentation:** Flutter widgets, провайдеры Riverpod, экраны.
- **Domain:** бизнес-логика в виде use cases, модели (сущности), интерфейсы репозиториев.
- **Data:** реализация репозиториев, работа с локальной БД (Drift), синхронизация (WebDAV), платформенные API (уведомления, контакты, secure storage).

### 2.2. Структура папок `lib/`

```
lib/
├── core/                          # Всё, что используется во всём проекте
│   ├── constants/                  # Константы (строки, числа, enum-подобные)
│   ├── utils/                      # Вспомогательные функции
│   ├── encryption/                  # Логика шифрования (AES-256)
│   └── extensions/                  # Расширения для встроенных типов (DateTime, String и т.д.)
├── data/                           # Реализация источников данных
│   ├── local/                       # Drift база данных, DAO, миграции
│   ├── sync/                        # WebDAV клиент, шифрование/дешифрование для синхронизации
│   └── platform/                     # Обёртки над системными API (уведомления, контакты, secure storage)
├── domain/                          # Бизнес-слой
│   ├── models/                       # Сущности (Event, Person, Reminder и т.д.)
│   ├── repositories/                  # Интерфейсы репозиториев (например, EventRepository)
│   └── usecases/                      # Классы-команды (AddEvent, GetUpcomingEvents, ...)
├── features/                         # Модули по фичам
│   ├── events/                        # Управление событиями
│   ├── reminders/                     # Настройки напоминаний
│   ├── notifications/                  # Логика уведомлений (шедулер, обработчик кликов)
│   ├── import_export/                  # Импорт/экспорт JSON, CSV
│   ├── sync/                           # Настройки синхронизации (WebDAV)
│   └── settings/                       # Настройки приложения
├── presentation/                     # Общие UI-компоненты, темы, локализация
│   ├── screens/                        # Экраны (могут переиспользовать фичи)
│   ├── widgets/                        # Переиспользуемые виджеты
│   ├── providers/                      # Глобальные Riverpod провайдеры
│   └── l10n/                           # Локализация (ARB файлы)
└── main.dart
```

**Важно:** Фичи могут иметь свою внутреннюю структуру (presentation/domain/data), если логика достаточно сложная. Например, `features/events/` может содержать:
- `events_provider.dart`
- `events_screen.dart`
- `event_form_widget.dart`
- `event_repository_impl.dart` (если реализация специфична для фичи, но лучше выносить в общий data)

### 2.3. Принципы проектирования
- **SOLID** – особенно принцип разделения интерфейсов и инверсии зависимостей.
- **DRY** – избегать дублирования кода.
- **KISS** – простота превыше всего.
- **YAGNI** – не добавлять функционал, который не нужен сейчас.

## 4. ПРАВИЛА КОДИРОВАНИЯ

### 4.1. Общие требования
- Код должен быть **читаемым** (понятные имена переменных, функций, классов).
- Комментарии **обязательны** для сложной логики, публичных API, use cases.
- Избегать магических чисел и строк – выносить в константы (в `core/constants`).
- Строгое соблюдение структуры папок.
- Использовать **type safety** – избегать `dynamic`, предпочитать конкретные типы.
- Обрабатывать ошибки: использовать `Either` (из `fpdart` или кастомный) или `Result` для возврата ошибок в use cases. Не бросать исключения в презентационном слое.

### 4.2. Именование
- **Файлы:** `snake_case.dart`
- **Классы:** `PascalCase`
- **Методы/функции:** `camelCase`
- **Константы:** `kCamelCase` (префикс `k`), например `kDefaultReminderDays`.
- **Провайдеры Riverpod:** `eventListProvider`, `selectedEventProvider` (camelCase).

### 4.3. Архитектурные ограничения
- **Запрещено** использовать `BuildContext` вне виджетов (например, в use cases).
- **Запрещено** помещать бизнес-логику в виджеты.
- **Запрещено** прямое обращение к `Platform` из domain слоя (использовать интерфейсы).
- **Запрещено** использование `Firebase` или любых облачных сервисов, кроме WebDAV.
- **Запрещено** добавление телеметрии, аналитики, краш-репортов.
- **Запрещён** хардкод строк – использовать ARB-файлы локализации.

### 4.4. Обработка зависимостей
- Использовать **внедрение зависимостей** через Riverpod (провайдеры) или конструкторы.
- Для репозиториев и use cases – `@riverpod` или простые `Provider`.
- Для платформенных сервисов – создавать абстракции в `domain` и реализацию в `data/platform`.

### 4.5. Работа с датами
- Использовать пакет `intl` для форматирования и парсинга.
- Все даты хранить в UTC в БД, преобразовывать в локальное время только для отображения.
- Корректно обрабатывать 29 февраля для ежегодных повторений.

### 4.6. Уведомления
- Планирование через `flutter_local_notifications`.
- На Android использовать `workmanager` для периодической перепланировки.
- Обязательно перепланировать уведомления после перезагрузки (слушать `BOOT_COMPLETED` через `android_alarm_manager_plus` или `workmanager`).

## 5. ПРИНЦИПЫ ПРИВАТНОСТИ И БЕЗОПАСНОСТИ

- Все данные хранятся локально. Единственное сетевое взаимодействие – опциональная WebDAV синхронизация (только по инициативе пользователя).
- Никакой телеметрии, краш-репортов, логирования в файлы.
- Ключи шифрования хранятся в `flutter_secure_storage`.
- Для экспорта/импорта используется AES-256 шифрование с паролем, задаваемым пользователем.
- Пароль пользователя никогда не сохраняется в открытом виде (только для сессии, если нужно).
- Весь код открыт, лицензия MIT.

## 6. ПРИМЕРЫ ПРОМПТОВ ДЛЯ AI

### 6.1. Генерация use case
```
Создай use class AddEvent в папке domain/usecases. Он должен принимать EventRepository и NotificationScheduler. Метод execute принимает EventModel и возвращает Future<void>. Валидация: дата не может быть в прошлом (если это не повторяющееся событие). После сохранения запланировать уведомление.
```

### 6.2. Генерация репозитория
```
Реализуй интерфейс EventRepository в data/repositories/event_repository_impl.dart. Используй Drift базу данных (database) и маппинг из Drift моделей в доменные модели.
```

### 6.3. Генерация экрана
```
Создай экран списка событий (events_screen.dart) с использованием Riverpod. Отображай список событий из eventListProvider. При нажатии на событие переходи на детали. Добавь FAB для создания нового события.
```
